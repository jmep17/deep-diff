# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Deep Diff (package name `deep-diff`) is an Electron + React desktop app for local visual regression testing. It lets a user pick a git repository, spin up a "sidecar" dev server for a given branch/worktree, scan the repo for API endpoints and frontend routes, configure mock profiles for those endpoints, and run a pixel-diff comparison between two refs (branches or the working tree) for the detected routes.

Node version is pinned via `.node-version` (managed by `fnm`); `scripts/ensure-node.sh` puts the right Node on `PATH` and is sourced by the other shell scripts in `scripts/`.

## Commands

Package manager is pnpm (`pnpm@10.28.2`, see `packageManager` field — use pnpm, not npm/yarn).

- `pnpm run dev` — runs the Vite renderer and Electron concurrently for local development (`dev:renderer` on port 5180 + `dev:electron`, which waits for the renderer then builds and launches Electron pointed at the dev server).
- To verify the running app's renderer/bridge over CDP: `pnpm run dev:renderer` (port 5180), then `env VITE_DEV_SERVER_URL=http://127.0.0.1:5180 pnpm exec electron . --remote-debugging-port=9222`, then `agent-browser connect 9222` + `agent-browser eval "Object.keys(window.deepDiff)"` / `screenshot`. agent-browser captures web contents only — **not** native macOS file dialogs. You therefore can't reach the live sidecar/webview over CDP — workspace selection is a native folder dialog and `authorizedRoots` is dialog-only (no env seed); exercise webview/overlay/probe logic with a standalone Electron harness driving `dist-electron/` directly (see `scripts/test-change-probe.cjs`, `scripts/test-mock-rendering.cjs`).
- `pnpm run build` — typechecks both the renderer and Electron main process, builds the renderer with Vite, then compiles Electron's TypeScript and bundles the preload to CommonJS (`build:electron`) to `dist-electron/`.
- `pnpm run typecheck` — runs `tsc -p tsconfig.json` (renderer, noEmit) and `tsc -p tsconfig.electron.json` (Electron main process) without building.
- `pnpm run start` — launches Electron against the already-built `dist/`/`dist-electron/` output (no dev server).
- `pnpm run test:e2e` — runs the full Cypress e2e suite headless (`scripts/test-e2e.sh`): starts a dedicated Vite dev server on port 5174 (override with `DEEP_DISH_E2E_PORT`) and runs Cypress against it via `start-server-and-test`. Browser defaults to `electron`; override with `CYPRESS_BROWSER`.
- `pnpm run test:e2e:open` — same setup but opens the interactive Cypress runner (`scripts/test-e2e-open.sh`).
- To run a single Cypress spec headless: `DEEP_DISH_E2E_PORT=5174 pnpm exec cypress run --browser electron --spec cypress/e2e/<file>.cy.ts` (or just use `cy:open` and pick the spec in the UI).
- `node scripts/validate-endpoint-mocks.mjs` — standalone script asserting the endpoint scanner's output shape/coverage against the `mock-repositories/auth0-routes-fixture` fixture. Requires `dist-electron/` to be built first (`pnpm run build:electron`).

Requires `dist-electron/` built first.
- `node scripts/test-mock-repository.mjs` — broader integration test of local-mode scanning, sidecar launch/worktree creation, GitHub remote-mode lookups (skipped without a `GITHUB_TOKEN` or `gh auth token`), and an end-to-end visual diff run via `scripts/electron-app` against the same fixture. Also requires `dist-electron/` to be built first.

- `node scripts/test-auth-config-detector.mjs` — asserts `electron/authConfigDetector.ts` detection logic across env files, config files, and package.json deps. Requires `dist-electron/` built first (`pnpm run build:electron`).
- `node scripts/test-scenario-repos.mjs` (or `pnpm run test:scenarios`) — drives a real headless visual diff (via `scripts/visual-diff-electron/run.cjs`, a parametrized Electron runner) against the `mock-workspace/` scenario repos and asserts Auth0 detection, endpoint/route scans, and that changed routes are flagged while control routes stay byte-identical (`mismatchRatio` 0). Requires `dist-electron/` built first. Verifies the diff _engine_ end-to-end, not the Electron GUI/IPC layer.

Asserts exact/`:param` overrides return mock bodies, non-matched routes + page HTML pass through to the real server, method discrimination holds, the no-override path still exposes the (pass-through) proxy URL, and that runtime discovery records the observed JSON endpoints (`getObservedEndpoints`) while ignoring the HTML page route. Requires `dist-electron/` built first.

Requires `dist-electron/` built first.
- `pnpm run test:change-probe` (`electron scripts/test-change-probe.cjs`) — runtime smoke test for the change-probe **collector** under real Electron/Chromium: loads a DOM with `data-dds-source` elements, runs the injected collector via `executeJavaScript`, and asserts probes (with real rects) flow through `buildChangeLinks`. Covers the collector→serialization→match seam the pure unit test can't. Requires `dist-electron/` built first.
- `pnpm run test:mock-rendering` (`electron scripts/test-mock-rendering.cjs`) — end-to-end check that an active endpoint mock **visibly changes the rendered page**. Launches the `storefront-auth0` sidecar with `endpointOverrides` (so it's fronted by the mock proxy), loads each control page in a real `BrowserWindow` (executing the client reconciler), and asserts the DOM reflects the mock (empty cart, Free-plan upgrade prompt, product count) — and that the no-override path shows the real server-rendered baseline. This is the only test of page _consumption_ of a mock; the scanner/proxy seams are covered by `test:sidecar-mocks`. Requires `dist-electron/` built first.
- `node scripts/test-repo-overlay.mjs` (or `pnpm run test:repo-overlay`) — plumbing test for **repo overlays** (`electron/repoOverlay.ts`): smokes `ensureOverlayScaffold` (folder + `OVERLAY-README.md`) and `applyOverlay` (README skipped), then launches the `storefront-auth0` sidecar on a feature branch with an `overlayDir` whose `server.mjs` serves a unique marker and asserts the worktree serves it (proves overlay files reach the worktree before spawn). Pure Node; requires `dist-electron/` built + `setup:fixtures`.
- `pnpm run test:auth-seam` (`electron scripts/test-auth-mock-seam.cjs`) — proves the overlay auth bypass _seam_: scaffolds a throwaway Vite + `@auth0/auth0-react` app with a fake Auth0 domain, confirms it redirects off-origin without the mock, and that aliasing the SDK → a local mock stops the redirect and renders the protected route under a hidden `BrowserWindow`. Standalone (no `dist-electron/`); **needs network** (installs react/vite/auth0-react into a temp dir).
- `node scripts/test-server-logs.mjs` (or `pnpm run test:server-logs`) — pure-Node test of **server-output capture** (`electron/serverLogs.ts`): `LogSink` unit (line-prefixed file writes, structured `entries[]`, `logBus` emit, `getLogDir()` env honoring, retention prune) + asserts `launchSidecar` returns `status.logFile` and the file captures the dev server's `[sidecar] [stdout]`. Can't cover the page-console seams (no Chromium / no renderer). Requires `dist-electron/` built + `setup:fixtures`.
- `pnpm run test:server-console` (`electron scripts/test-server-console.cjs`) — real-Chromium test of the **diff page-console** seam (`attachConsoleCapture`: a `data:` page's `console.log`/`console.error` → `[base] [console…]` in the sink) **and** a real `runVisualDiff` asserting `report.logFile` has both `[base]` and `[target]` `[stdout]` lines + a populated `report.logs` snapshot (the diff path imports `electron`, so it can only run under Electron). The sidecar webview→`logs:append` console path is covered only by the app / manual agent-browser. Requires `dist-electron/` built + `setup:fixtures`.
- `bash scripts/with-node.sh <command>` — runs an arbitrary command with the pinned Node version on `PATH` (useful in shells without `fnm` auto-loading).

There is no unit test runner configured (no Jest/Vitest) — correctness is validated via Cypress e2e specs (`cypress/e2e/`) and the Node integration scripts above, which exercise the real `electron/` modules against the `mock-repositories/auth0-routes-fixture` fixture repo.

## Architecture

### Process split: `electron/` vs `src/`

This is a standard Electron app with two separate TypeScript builds (see the two tsconfigs):

- `electron/` (main process, Node context, compiled by `tsconfig.electron.json` → `dist-electron/`) does all filesystem/git/process work: scanning workspaces for git repos, listing branches, talking to the GitHub REST API, detecting frontend routes and API endpoints by walking source files with regex/heuristics (no AST parsing), spawning sidecar dev servers (optionally in a throwaway `git worktree` for non-current branches), and running the visual diff (spawns two dev servers, loads each route in a hidden `BrowserWindow`, captures bitmaps, and diffs pixel-by-pixel with a hardcoded threshold).
- `src/` (renderer, browser context, compiled by `tsconfig.json` via Vite) is the React UI. It is currently a single large component tree in `src/App.tsx` (~1000 lines, no further breakdown into `src/components/`) backed by `src/data/seed.ts` for placeholder/demo data and `src/lib/types.ts` for shared renderer-side types.
- `electron/preload.ts` bridges the two via `contextBridge`, exposing a `window.deepDiff` API (typed in `src/vite-env.d.ts`) with `contextIsolation: true` / `nodeIntegration: false`. All renderer → main-process calls go through this bridge and `ipcMain.handle` registrations in `electron/main.ts`. **Preload must ship as CommonJS**: `package.json` is `"type": "module"`, so `tsc` emits `.js` as ESM, which a sandboxed Electron preload cannot load. `build:electron` therefore bundles the preload to `dist-electron/preload.cjs` (esbuild, `--format=cjs --external:electron`) and deletes the stale `preload.js`; `main.ts` loads `preload.cjs`. If the preload is ESM, Electron logs `Unable to load preload script`, `window.deepDiff` is `undefined`, and the renderer silently falls back to demo data (`App.tsx` `if (!bridge)`).
- **Type duplication**: `electron/types.ts` and `src/lib/types.ts` independently define overlapping interfaces (`RepositorySummary`, `EndpointDefinition`, `VisualDiffReport`, etc.) because the two processes have separate compiler configs/module resolution and can't share imports across the main/renderer boundary. When changing one of these shared shapes, update both files and the `window.deepDiff` signatures in `src/vite-env.d.ts` + `electron/preload.ts` together.

### Key main-process modules

- `electron/repositories.ts` — discovers local git repos under a workspace folder, lists local branches, and fetches GitHub repos/branches via the REST API (token optional, passed per-request from the renderer).
- `electron/routeDetection.ts` — heuristically maps Next.js `app/`/`pages/` files to frontend routes (handles dynamic segments like `[id]`, route groups like `(marketing)`, and produces both a display path and a sample `urlPath` with fixture values substituted for dynamic segments). Also used to figure out which routes are impacted by a set of changed files (`detectVisualRoutesForChangedFiles`), treating changes under `src/`, `middleware.*`, `.env*`, or non-page/non-API source files as having "shared impact" on all routes.
- `electron/endpointScanner.ts` — heuristically detects API endpoints from Express/Fastify-style route definitions, Next.js App Router `route.ts` exports, Next.js Pages API files, and OpenAPI/Swagger JSON specs; **and** from outbound API _calls_ the app makes (`detectApiCalls`): `fetch('/x')`, `axios('/x',{method})`, `axios.get('/x')`, and generic `IDENT.method('/x')` on **any** receiver (not just `app|router|server|fastify`). It infers field shapes/mock values from nearby object-literal source text for definitions (not real parsing); call sites get an **empty** source slice so they fall back to the generic `{id,status}` mock instead of scraping caller-side keys. Call detection is gated on the URL literal being path-like (starts with `/` or `http(s)://`) plus a receiver denylist (`map`, `cookies`, `searchParams`, `request`/supertest, …) and skips test files, which together suppress `map.get('key')`-style false positives. Template `${id}` → `:param`; query/hash stripped; absolute URLs keep their pathname, are labelled `external call (host)` at **low** confidence (the same-origin proxy can't intercept cross-origin — that's Phase-4 interceptor territory). Dedup is by `METHOD:path` keeping the **highest-confidence** detection, so a real route definition (high) always beats a duplicate call site (medium/low). **Still source-file driven** in that only _literal_ URLs are reachable here — URLs built from variables/`baseURL`/templates are caught by runtime discovery (the always-on proxy), not the static scan; and like any regex-on-text scan, example calls in comments/strings can false-positive. A `server.mjs` runtime route with no source file and no call site remains invisible to the static scan.
- `electron/sidecar.ts` / `electron/visualDiff.ts` — both infer how to run a target repo (reads `package.json` scripts/`packageManager`, prefers `dev` then `start`), allocate a free local port, and spawn the dev server. For any ref other than the special sentinel `__working_tree__`, they create a detached `git worktree` in the OS temp dir so the base/target comparison can run two versions of the repo simultaneously without touching the user's checkout; the worktree is removed in a `cleanup()` once the server stops. `visualDiff.ts` additionally drives a hidden `BrowserWindow` to capture each route on both servers and builds a diff image via manual pixel comparison (`diffThreshold = 18` per RGB channel sum). **Cleanup is fire-and-forget** (`void …cleanup()`), so a headless caller that exits right after a run leaks worktrees + dev servers — prune in non-GUI runners (see `scripts/test-scenario-repos.mjs` `pruneWorktrees`). Also `SidecarStatus.pid` is the `pnpm`/shell **wrapper**; the listening `node server.mjs` is a grandchild, so killing `status.pid` won't free the port (use `pkill -f 'server.mjs --'`). `launchSidecar` **always** fronts the dev server with the proxy (even with zero overrides — pure pass-through), so `status.url` is **always** the proxy (which accepts connections before the server it fronts is listening → `Proxy error: ECONNREFUSED`); poll `http://127.0.0.1:${status.port}` (the real server) for readiness, not `status.url`. The proxy reads a **mutable** module-level override map, so `setSidecarOverrides(overrides)` updates a running sidecar in place without a relaunch and **without any URL change** (`sidecar:setOverrides` IPC) — the proxy is already up, so the first override is just served at the same `status.url`; an empty map turns a mock back off (pass-through). The renderer (`SidecarPanel`) pushes the active profile's effective overrides on every toolbar toggle, then reloads the `<webview>` so the page re-fetches through the proxy. The always-on proxy also performs **runtime endpoint discovery**: it records every JSON response (and every mock-served request), keyed `METHOD:path` with concrete dynamic segments collapsed to `:id` (page navigations + static assets ignored), and emits new ones on the exported `discoveryBus` → `main.ts` forwards them to the main window via `webContents.send('endpoints:observed', …)` → the renderer merges them into the (mockable) endpoint inventory tagged `framework: 'observed (runtime)'`. This catches dynamic same-origin URLs (variables/`baseURL`/templates) that the static `endpointScanner` can't see; `getObservedEndpoints()` exposes the current set for non-GUI callers/tests. Both now capture the **full** dev-server stdout/stderr + install output into a per-run `LogSink` (see `electron/serverLogs.ts`): `launchSidecar` returns `status.logFile`; `runVisualDiff` labels lines `[base]`/`[target]` and returns `report.logFile` + a bounded `report.logs` snapshot. On a startup failure both still **throw**, but the message now carries `(logs: <file>)`.
- `electron/serverLogs.ts` — **server/page log capture**. A `LogSink` (one per launch / diff run) writes every line to a per-run file under `getLogDir()` (`DEEP_DISH_LOG_DIR`, set by `main.ts` to `userData/logs`; falls back to `os.tmpdir()` for direct-call scripts) prefixed `[server] [stream]`, keeps a capped in-memory `entries[]` snapshot, prunes oldest run logs beyond a cap, and re-emits each line on the exported `logBus`. `main.ts` subscribes to `logBus` and forwards entries to the **main window only** via `webContents.send('logs:event', …)` (the app's only main→renderer stream) — the renderer's `LogDrawer` (`src/App.tsx`) renders them. `attachConsoleCapture(webContents, getServer, sink)` folds a page's browser console into the sink: `visualDiff.ts` captures the hidden capture window directly; the live sidecar preview page lives in the renderer `<webview>`, so its console is forwarded over the `logs:append` IPC → `appendSidecarConsole`. `logs:reveal` opens a run's log file in the OS file manager (path confined to `getLogDir()` via `realpath`). The module is Electron-free so the integration scripts can drive the core modules directly.
- Set `DEEP_DISH_DIFF_TRACE=1` to get a timestamped trace log at `electron/visual-diff.trace` (or `dist-electron/visual-diff.trace` once built) from `visualDiff.ts`'s `trace()` helper — useful for debugging hangs in the dev-server-start/capture/cleanup pipeline.
- `electron/authConfigDetector.ts` — detects whether a target repo uses Auth0 by scanning `.env*` files for `AUTH0_BASE_URL`/`APP_BASE_URL`/`AUTH0_ISSUER_BASE_URL`, checking for `auth0.config.{js,ts}`, and inspecting `package.json` deps. Used by `sidecar.ts` and `visualDiff.ts` to inject corrected `AUTH0_BASE_URL`/`APP_BASE_URL` env vars at spawn time (matching the dynamically allocated port). **Gotcha for visual-diff fixtures:** that injected port differs between the base and target sidecars, so any _captured page_ that renders the injected base URL will false-positive — keep port-dependent values out of page HTML (expose them only in non-captured API JSON).
- `electron/changeLink.ts` — core of the **code-change → on-page element linking** feature. `getChangedFiles(repoPath, baseRef, targetRef)` runs `git diff --name-only` between two refs (a ref of `__working_tree__` diffs against the working tree). `cleanSourcePath`/`matchElementSource`/`buildChangeLinks` take a raw source string read off a rendered DOM element and decide whether it originates from a changed file (reuses `normalizeRelativePath` from `routeDetection.ts`). The renderer (`SidecarPanel` in `src/App.tsx`) injects `CHANGE_PROBE_SCRIPT` into the sidecar `<webview>` via `executeJavaScript` to collect element sources, sends them over the `changes:link` IPC, and draws `.change-overlay` highlight boxes (coords are guest-viewport-relative, which align because `.preview-webview` and `.change-overlays` are both `inset:0` of `.pizza-page`). **Source-extraction caveat:** the collector reads source from `data-dds-source`/`data-source`/`data-inspector-relative-path` attributes (React-version-independent, the reliable path) _or_ React dev-fiber `_debugSource` — but `_debugSource` was **removed in React 19**, so on a stock modern Next.js dev target only the data-attribute path fires. Verified by `test:change-link` (pure core) + `test:change-probe` (collector in real Chromium). **Webview readiness:** `SidecarPanel` hides the `<webview>` (`.preview-webview.loading`) until its first _genuine_ load — `did-stop-loading` with **no** preceding `did-fail-load` that navigation — because `did-fail-load`, `dom-ready`, AND `did-finish-load` all fire for Chromium's connection-refused error page (which flashes during sidecar startup before the dev server listens). Do not gate "loaded" on `dom-ready`/`did-finish-load`.
- `electron/repoOverlay.ts` — **per-repo overlay files** that bypass things the capture can't handle as-is (canonical case: a client-side `@auth0/auth0-react` redirect that stops routes ever rendering). The user drops files (e.g. a `vite.config.ts` aliasing the auth SDK to a mock + the mock) into a per-repo folder **in Deep Diff's own storage** — `<DEEP_DISH_OVERLAY_ROOT or userData/overlays>/<repo-basename>-<8charhash>/`; `applyOverlay` copies them over each throwaway worktree (mirroring repo root, whole-file overwrite) inside `prepareRuntimeRepository` **after `git worktree add`, before install**, in both `sidecar.ts` and `visualDiff.ts`. The target repo's git history is never touched and the files never reach the user's own dev/prod (worktree-only). `main.ts` resolves the root and sets `request.overlayDir` **after** validation (never renderer-supplied — it's a copy source, kept off the IPC surface); `ensureOverlayScaffold` writes an `OVERLAY-README.md` (reserved name, never copied) into the folder documenting exact location/shape. Full spec: `docs/repo-overlays.md`. Seam proven by `test:auth-seam`, plumbing by `test:repo-overlay`. Caveats: whole-file overwrite (no merge); applies to worktree refs only (working-tree-vs-branch overlays one side); mock stops the redirect but pages still need endpoint mocks for content.
- `electron/ipcValidation.ts` / `electron/logger.ts` — `main.ts` validates every renderer-supplied IPC payload before delegating (path must resolve via `fs.realpath` to a directory under a user-authorized workspace root tracked in `authorizedRoots`; `command` overrides are allowlisted to known runners; git refs reject leading `-`). All handlers are wrapped in `registerHandler` (logs + re-throws). **Keep this validation at the handler layer only** — the Cypress mock bridge and the `scripts/*.mjs` integration scripts call the core functions (`runVisualDiff`, `launchSidecar`, `scanEndpoints`) directly from `dist-electron/` and bypass it, so registry-dependent checks must not move into those core modules.

### Test fixture: `mock-repositories/auth0-routes-fixture`

A self-contained mock Next.js-ish app, committed as its own nested git repository (it has its own `.git/`), used as the ground truth for both Cypress specs (via tasks registered in `cypress.config.ts`) and the Node integration scripts. It intentionally mixes route styles (Next.js App Router API routes, Pages API routes, Express-style routes, OpenAPI, and frontend pages) and has two branches (`main` and `feature/auth0-preview-callbacks`) with known, documented differences. `fixture-expectations.json` in that fixture is the source of truth for expected endpoint counts/paths and expected visual-diff results (changed vs. unchanged routes) — `mock-repositories/auth0-routes-fixture/README.md` documents the expected comparison table. Don't edit this fixture casually; scanner/route-detection changes should be checked against its expectations. The fixture has both `.env.example` (template) and `.env` (active, with `AUTH0_BASE_URL=http://localhost:3000`) — the `.env` is intentional and used by `authConfigDetector.ts` tests.

### Scenario repos: `mock-workspace/`

`mock-workspace/` is a second, separate workspace folder (so `scanWorkspace` discovers it as a multi-repo workspace, unlike the single-repo `mock-repositories/`). It holds two **real-diff** scenario repos, each its own nested git repo with a `main` and a feature branch whose differences are **actual committed source changes** (not the `DEEP_DISH_DIFF_BRANCH` env trick the canonical fixture uses), so the worktree-based visual diff is driven by genuinely different code:

**Clone-ability / restore:** these are nested git repos, so (like `mock-repositories/auth0-routes-fixture`) their live directories are **gitignored** and their full history — both branches — ships as a committed sibling git bundle (`mock-workspace/<name>.bundle`). A fresh clone of the parent repo has empty `mock-workspace/<name>/` dirs until you run **`pnpm run setup:fixtures`** (`scripts/restore-fixtures.sh`, idempotent — clones each bundle, creates the local feature branch, checks out `main`). The scenario/change-link/mock-rendering tests fail with "no such ref" until this is run. **When you edit a scenario repo** (commit on both branches per the rules below), you must also **regenerate its bundle** so the change ships: `git -C mock-workspace/<name> bundle create "$PWD/mock-workspace/<name>.bundle" --all`, then commit the updated `.bundle` in the parent repo. The committed bundle — not the gitignored working dir — is the source of truth a fresh clone sees.

- `storefront-auth0` — realistic Auth0 config (`.env` AUTH0\_\* keys, `auth0.config.ts`, `lib/auth0.ts`, `@auth0/nextjs-auth0` dep, `middleware.ts`); `main` vs `feature/holiday-storefront-redesign` changes `/` and `/products/:productId`, leaving `/products`, `/account`, `/cart` byte-identical controls.
- `marketing-site` — no Auth0 (exercises the `authConfigDetector` negative path; a `.env` exists but has no AUTH0\_\* keys); `main` vs `feature/2026-brand-refresh` changes `/` and `/pricing`, leaving `/about`, `/contact` controls.

Both are zero-dependency (`dev` = `node server.mjs`) so they need no install. When a worktree _does_ have dependencies, `electron/installDependencies.ts` runs an install (frozen lockfile, falling back to plain) after `git worktree add` and before the dev server spawns (`prepareRuntimeRepository` in both `sidecar.ts` and `visualDiff.ts`); it short-circuits when no deps are declared or `node_modules` already exists, so these zero-dep fixtures stay a no-op. Each has a `fixture-expectations.json` + `README.md`. Validate with `pnpm run test:scenarios`. Determinism rule for these repos: page HTML must have no timestamps, randomness, or port-derived content, or control routes would diff against themselves across the two sidecars (default `mismatchTolerance` is 0).

**Endpoint-driven conditional rendering (so mocks are visible):** several control pages fetch a GET endpoint client-side and re-render _only when the response differs from a server-rendered baseline embedded in the page_ (`mockReconciler` in each `server.mjs`). With no mock active the fetch equals the baseline so the DOM is untouched — that's what keeps control routes byte-identical for the visual diff. When a sidecar runs with `endpointOverrides` (fronted by the proxy), the response differs and the page visibly re-renders. Storefront: `/cart`←`/api/cart`, `/products`←`/api/products`, `/account`←`/api/auth/me`. Marketing: `/about`←`/api/team`. The embedded baseline must exclude port-dependent fields (e.g. the Auth0 `baseUrl`) via the reconciler's `transform`. Validate page _consumption_ of a mock with `pnpm run test:mock-rendering`. Note both repos tag every page `<main>` with `data-dds-source` so the change-element highlighter has elements to match — but it only highlights where that source file is in the `base..target` diff (storefront's & marketing's changed `.tsx` pages do differ across branches; the control pages' sources do not). Editing these repos: commit on **both** branches (the worktree engine reads committed refs, not the working tree), keep control/shared code byte-identical across branches, and verify with `git diff <base> <feature> -- server.mjs` (only the intended changed pages should differ).

## Notion task board

Project tasks for Deep Diff are tracked on a Notion Kanban board, accessed via the Notion MCP server. The board (a database titled "Tasks") lives under a "Deep Diff" page in the workspace. Its schema: `Status` (Not started / Planning / Ready / In progress / Done), `Priority` (High / Medium / Low), `Agent status` (free text, for an agent to report current activity), `Agent blocked` (checkbox, for an agent to flag when it needs user input), plus `Assignee` and `Due date`. Use `/notion:tasks:plan` and `/notion:tasks:build` to plan and execute tasks pulled from this board.

The `notion-query-data-sources` (SQL/view) and `notion-query-database-view` MCP tools require a paid Notion plan and return a `validation_error` on this workspace. To enumerate board tickets, use `notion-search` with the data source URL (`collection://d1693c53-799a-8208-b3be-07c471277659`) and then `notion-fetch` each page to read its `Status`/`Priority` properties.
